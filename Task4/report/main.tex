\section{Dataset Preparation}
\subsection{Label Aggregation}

We aggregate the annotation array [T, C, A] into a binary label matrix [T, C] using the \emph{get\_features\_and\_targets} function from the tutorial. The function first binarizes the overlap fractions with (annotations > 0).astype(int), so any nonzero overlap counts as a yes vote from that annotator.

The votes are then summed along the annotator axis and thresholded with \emph{votes > (A // 2)}, where A is the number of annotators in the file. This requires a strict majority — for A=2 both annotators must agree, for A=1 that single vote decides, and for higher A more than half must agree. We use the "multilabel" mode because multiple sound events can overlap in the same segment.

Counting any overlap as a yes vote is forgiving on segment boundaries, which matters since sound events rarely line up perfectly with the 1 second windows. The majority vote filters out individual annotator mistakes, like the missing keyboard\_typing we found in 000278.wav in the previous task.

We are biased towards false negatives — on two-annotator files a single disagreement drops the label, which especially hurts classes with low Jaccard agreement like door\_open\_close (52.3\%) and wardrobe\_drawer\_open\_close (51.9\%) (as seen in Task 3 of the project). Around 17\% of files only have one annotator, so their labels are never checked. Binarizing also throws away the overlap fraction — 5\% overlap and 95\% overlap end up treated the same.

An alternative would be taking the union of annotators, but that would just swap our false negatives for false positives. One could also weight the confidence in the annotator, for example if one of the annotators is also the author, we could choose to trust the author upon disagreement.

\subsection{Data Split}\label{sec:data-split}

(i) Verbatim to the description of data leakage in the tutorial: \emph{``Data leakage occurs when a model has access to information during training that would not be available at prediction time, leading to overly optimistic performance.''}
Consecutive frames have similar feature characteristics (high correlation), if these consecutive samples land in the validation and training data, we will get a misleadingly high performance for the training of the model, although it performs poorly on new or unseen data.

Originally the data is split randomly using the \emph{train\_test\_split()} method, for which data leakage is likely to occur. This is then improved, by selecting a whole adjacent section of the feature data array. This mitigates the information leakage effect, but still a large portion of our data is used for validation and not training.

To get a higher training effect from the data which is not used for testing, $k$-fold cross validation is used. This method sections the data which is not used for testing into $k$ equal sets, and selects one as the validation set and the others as training sets. This is done $k$ times while each iteration uses another validation set. Finally, the performance is averaged from iterations.

(ii) Equal class distribution is crucial, because a training set, which largely consists of a class which is not present in the validation set at all, would bring down performance. To ensure the classes are equally distributed across the $k$ different data splits, a method called \emph{stratified} $k$-fold cross validation is used \cite{k-fold}. A comparison of how the data is split can be seen in \autoref{fig:k-fold}. The method is later used implicitly by the builtin sklearn evaluation methods. The final split of the data is $90\%$ training + cross validation data and $10\%$ test data.

\subsection{Preprocessing}

Loudness or energy can vary between audio files, so normalizing these features might help. The simplest way to standardize the features, is by removing the mean and scaling them to unit variance. This can be achieved by applying the \emph{StandarScaler} from the sklearn. It is important that the same pre processing has to be applied when using the classifier in action.

\subsection{Evaluation Criterion}

From the previous task (Task 3) one could see, that the classes are strongly imbalanced. To evaluate the performance of a classifier, a list of metrics shown in \autoref{tab:metrics} can be taken into account.

& Description      & How many actual positives were caught. \\
& When it's useful & Critical when \emph{false negatives} are costly (e.g., medical diagnosis). \\
& Viable metric?   & \textbf{No}, prefers to predict more positives overall, implying more false positives (low threshold). Especially for multi-label classification it would be very likely for multiple false positives at once. \\
\midrule
\textbf{F1 Score}
& Description      & Harmonic mean of precision and recall. \\
& When it's useful & Good balance when classes are imbalanced. \\
& Viable metric?   & \textbf{Yes}, a balanced threshold is preferable here. On one hand we do not want the audio event detection to classify \emph{everything} as a label, but neither no events at all. \\

\midrule

\textbf{PR AUC}

A simple baseline classifier could be defined to predict the majority class. By extending the \emph{BaselineClassifier} for Multiple outputs, we realize that the majority class over all audio event is actually, that no class is predicted at all. The scores of this classifier are shown in \autoref{tab:scores-maj}. For the baseline metrics, the whole dataset has been taken into account.

\subsubsection{Empirical Class Frequency Classifier}\label{sec:empirical}

\begin{wrapfigure}[14]{r}{0.5\textwidth}
    \centering

    \label{fig:freq}

 The task description suggests taking class frequencies into account and make predictions based on this. For this, the baseline classifier has been extended. The classifier does not just predict the majority class, but takes the mean of how often each class occurs. Upon prediction, a class has a probability of occurring in the binary output vector based on frequency in the training set. \autoref{fig:freq} shows the probabilities for each class to be predicted from a sample in the validation set. The red line indicates where the original majority class baseline classifier would have sampled for a prediction; so in this case all zeros. The scores of this classifier are shown in \autoref{tab:scores-freq}.

Let these results be the baseline performance to compare other classifiers with. 

\section{Experiments}

The performance criterion to optimize the hyper parameters is chosen to be the \emph{F1 score}. For the Experiments, the \emph{Random Forest} classifier and a \emph{Light Gradient Boosting Machine} were selected. To evaluate which parameters perform best, a grid search was used to iterate over the combination of hyper parameters of each classifier\cite{gridsearch}. The data is split into $90\%$ training data and $10\%$ test data. The grid search performs a stratified\footnote{\emph{Stratified} meaning the cross validation is adapted to the class frequency. Better for imbalanced data as mentioned in \nameref{sec:data-split}} k-fold cross validation on the training data, therefor not manual split for the validation data was necessary. $k$ was chosen to be $5$.

\subsection{Random Forest Classifier}

This classifier was chosen, because it was suggested as a task in the classifier tutorial. It was reasonably fast to train, even on the whole dataset, which simplified the procedure. The most relevant hyper parameters of the random forest classifier are defined as follows:

\begin{itemize}[nolistsep,noitemsep]
    \item \textbf{n\_estimators}: Number of trees in the forest.
    \item \textbf{max\_depth}: Maximum depth of a tree.
    \item \textbf{min\_samples\_split}: The minimum number of samples required to split an internal node
\end{itemize}

To systematically find the optimal hyperparameter vector, $20\%$ of the dataset were used for the grid search. How the performance changes in the hyperspace can be seen in \autoref{fig:rf-hyp-space}. The maximum training F1 score is $0.258$, for the hyperparameter vector:
\textbf{n\_estimators} $=5$,
\textbf{min\_samples\_split} $=5$,
\textbf{max\_depth} $=20$.

\begin{figure}[ht]
    \centering
    \includegraphics[width=\linewidth]{rf-hyp-tuning}
    \caption{Hyperparameter Space of the random forest classifier. The darkest hyper parameter vector is chosen as the best model}
    \label{fig:rf-hyp-space}
\end{figure}

Following these trends, a higher max\_depth would indicate an even better performance. However, the training would be even more computationally expensive, so these parameters were accepted. It is interesting, that a lower number of total estimators perform better, which could be reasoned woth overfitting.

The determined parameters were then used to train the random forest classifier with the whole dataset. The prediction score on the unbiased test set is shown in \autoref{tab:scores-rf}. A significant improvement can be seen in relation to the baseline performance.

\subsection{Gradient Boosting Machine}

Initially, we tried out Support Vector Machines, however a Non-Linear SVM took to long to train and the linear SVM performed very poorly, which is why we tried the Gradient Boosting Machine (GBM)\cite{lgbm} in this section. The most relevant hyper parameters of the GBM classifier are defined as follows:

\begin{itemize}[nolistsep,noitemsep]
    \item \textbf{n\_estimators}: Number of boosted trees to fit.
    \item \textbf{max\_depth}:  Maximum tree depth for base learners.
    \item \textbf{learning\_rate}: The minimum number of samples required to split an internal node
\end{itemize}

To systematically find the optimal hyperparameter vector, $20\%$ of the dataset were used for the grid search. How the performance changes in the hyperspace can be seen in \autoref{fig:rf-hyp-space}. The maximum training F1 score is $0.526$, for the hyperparameter vector:
\textbf{n\_estimators} $=100$,
\textbf{learning\_rate} $=0.1$,
\textbf{max\_depth} $=30$.

As expected, a higher number of estimators as well as deeper trees perform better.
